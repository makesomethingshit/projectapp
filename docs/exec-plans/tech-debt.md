# 기술 부채

## Harness 점검 중 발견한 문제

- 초기 점검 시 기존 문서와 일부 UI 문자열에 인코딩 손상으로 보이는 깨진 텍스트가 있었습니다. `README.md`, `gemini.md`, `docs/APP_STRUCTURE.md`, `docs/DESIGN_SYSTEM.md`, `docs/DEPENDENCIES.md`, `docs/codex.md`는 새 harness 문서로 연결되는 한국어 호환 문서로 정리했습니다. 앱 소스 문자열과 `CHANGELOG.md`에는 여전히 깨진 텍스트가 남아 있습니다.
- `package.json`에 `lint`, `typecheck`, `test`, `build` 스크립트가 없습니다. 테스트는 루트의 독립 실행 파일을 직접 실행해야 합니다.
- `app.js`와 `app-graph-events.js`가 많은 책임을 가진 큰 조율 파일입니다.
- `appSettings`에 사용자 설정, 그래프 레이아웃, 수식 노드, 아카이브 그래프 노드, 히스토리, 활동 로그, 단축키가 함께 들어 있습니다.
- 제품 맥락에는 폴더가 중요하지만 현재 코드에는 별도 Folder 모델이 없습니다.
- 프로젝트 `status`는 수동 저장값이라 진행도, 마감, 작업 상태와 어긋날 수 있습니다.
- 프로젝트 `progress`와 `advance`는 저장되지만 실제 표시값은 작업, 하위 프로젝트, 외부 링크, 수식에서 파생될 수 있습니다.
- 완성도 롤업은 가중치를 사용하고 진행도 롤업은 평균을 사용합니다. 이 비대칭이 의도인지 확인이 필요합니다.
- 하위 프로젝트가 완성도 기여자로 존재하면 직접 작업이 상위 프로젝트 완성도에 반영되지 않을 수 있습니다.
- `test_build_data.js`는 assert 중심 테스트라기보다 출력 기반 진단 스크립트에 가깝습니다.
- 작업 폴더 안에 `dist`, `node_modules`, 추출 바이너리 같은 생성물과 의존성 폴더가 함께 있어 저장소 지도가 쉽게 복잡해집니다.

## 지금은 고치지 말 것

위 항목은 문서화된 발견 사항입니다. 동작 변경은 별도 실행 계획으로 분리해 의도와 테스트를 명확히 한 뒤 진행합니다.
